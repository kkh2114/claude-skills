# 코드 리뷰 루프 참조 v1

> "사람이 코드를 한 줄씩 읽어야 할 필요가 줄어든다.
>  사람의 일은 이 방어 체계 자체를 설계하고 개선하는 것이다." — OpenAI

## Ralph Wiggum Loop

에이전트가 코드를 작성 → 자체 리뷰 → 다른 에이전트에게 리뷰 요청 →
피드백 반영 → 반복 → 모든 리뷰어 만족 시 완료.

```
┌─────────────────────────────────────┐
│  1. 코딩 에이전트: 코드 작성         │
│     ↓                               │
│  2. 자체 로컬 검증                   │
│     - 린터 실행                      │
│     - 구조 테스트 실행               │
│     - 유닛 테스트 실행               │
│     ↓                               │
│  3. PR 생성 (또는 리뷰 요청)         │
│     ↓                               │
│  4. 리뷰 에이전트 검토               │
│     - 아키텍처 준수?                 │
│     - 테스트 충분?                   │
│     - 컨벤션 준수?                   │
│     - 보안 이슈?                     │
│     ↓                               │
│  5. 피드백 수신                      │
│     ├─ 통과 → 머지 진행             │
│     └─ 수정 필요 → 2로 돌아감       │
│                                     │
│  ※ 대부분의 "nitpicking"은          │
│    에이전트 간에 처리된다.           │
│    사람은 진짜 판단이 필요할 때만.   │
└─────────────────────────────────────┘
```

## 에이전트 자율성 3단계

### 레벨 1: 사람 최종 승인

```
에이전트 작성 → 에이전트 리뷰 → 사람 리뷰 → 머지
```
- 모든 PR에 사람이 최종 확인
- 초기 프로젝트, 높은 리스크 단계에 적합
- 사람이 병목이 될 수 있음

### 레벨 2: 에이전트 자동 머지 + 사람 감독

```
에이전트 작성 → 에이전트 리뷰 → 자동 머지
                                  ↓
                            사람에게 알림
                            (에스컬레이션 시만 개입)
```
- 에이전트 리뷰 통과 + CI 통과 시 자동 머지
- 사람은 대시보드/알림으로 모니터링
- 에스컬레이션 기준 명확히 정의 필요

### 레벨 3: 완전 자율

```
에이전트 작성 → 에이전트 리뷰 → 자동 머지 → 자동 배포
                                              ↓
                                        모니터링 + 롤백
```
- 고도의 테스트 커버리지와 관찰 가능성 필요
- 이상 감지 시 자동 롤백
- 현재는 소수의 팀만 도달한 수준

## 사람 에스컬레이션 기준

다음 경우 에이전트가 사람에게 넘겨야 한다:

| 상황 | 이유 |
|------|------|
| 새로운 아키텍처 결정 | 장기적 영향, 에이전트 판단 범위 밖 |
| 보안 관련 변경 | 인증, 암호화, 권한 변경 |
| 외부 API/서비스 연동 변경 | 비용, 계약, SLA 영향 |
| 제품 방향 판단 | 비즈니스 결정 |
| 에이전트 간 의견 불일치 | 해결 불가한 트레이드오프 |
| 데이터 마이그레이션 | 되돌리기 어려운 변경 |

## 리뷰 에이전트 프롬프트 포인트

리뷰 에이전트가 체크하는 관점:

```markdown
## 리뷰 체크리스트
1. 아키텍처: 레이어 규칙 준수? 모듈 경계 침범 없음?
2. 테스트: 새 코드에 테스트 있음? 기존 테스트 깨지지 않음?
3. 컨벤션: 네이밍, 구조, 패턴이 docs/conventions.md 준수?
4. 보안: 입력 검증? 인증/인가 체크? 민감 데이터 노출?
5. 성능: 불필요한 쿼리? N+1 문제? 메모리 누수 가능성?
6. 문서: 관련 docs/ 업데이트 필요? API 변경 시 설계 문서 갱신?
```

## docs/review-protocol.md 생성 가이드

Generator가 만들 때 포함할 내용:
- 위 체크리스트를 프로젝트에 맞게 커스터마이징
- 자율성 레벨 (사용자가 Phase 1에서 선택한 것)
- 에스컬레이션 기준 구체화
- 리뷰 통과 조건 (CI 전체 통과 + 리뷰어 승인)
