# docs/ 디렉터리 구조 가이드 v1

> docs/는 에이전트의 "유일한 진실의 원천(single source of truth)"이다.
> 에이전트 관점에서, 인-컨텍스트로 접근할 수 없는 것은 존재하지 않는다.

## 필수 파일

### architecture.md
레이어 구조, 모듈 경계, 의존성 방향 규칙.
에이전트가 새 코드를 작성할 때 "어디에 놓을지" 판단하는 근거.

포함 내용:
- 레이어 다이어그램 (ASCII 또는 Mermaid)
- 의존성 방향 규칙 (정방향만 허용)
- 도메인 목록과 각 도메인의 책임
- 횡단 관심사 처리 방식 (Providers)
- 금지 패턴 (예: UI에서 DB 직접 접근)

### conventions.md
코딩 스타일, 네이밍, 파일 구조 규칙.

포함 내용:
- 파일/폴더 네이밍 규칙
- 함수/변수 네이밍 규칙
- import 순서
- 에러 처리 패턴
- 로깅 패턴
- 선호하는 라이브러리 (+ 금지 라이브러리)

### quality-grades.md
도메인/레이어별 현재 품질 등급.

```markdown
## 품질 등급 현황

| 도메인 | 등급 | 테스트 커버리지 | 기술 부채 | 비고 |
|--------|------|---------------|----------|------|
| auth | A | 92% | 낮음 | |
| billing | B | 78% | 중간 | 결제 연동 리팩터 필요 |
| chat | C | 45% | 높음 | 테스트 보강 우선 |

최종 업데이트: YYYY-MM-DD
```

에이전트가 주기적으로 업데이트. 품질이 떨어지면 리팩터링 우선순위 상승.

### testing-strategy.md
테스트 유형, 커버리지 기준, 실행 방법.

포함 내용:
- 유닛 테스트: 각 도메인 __tests__/ 디렉터리, 커버리지 기준
- 통합 테스트: tests/integration/, API 테스트 방법
- E2E 테스트: tests/e2e/, 브라우저 자동화 도구
- 테스트 실행 커맨드
- "기능 완료" 판정 기준 (어떤 테스트가 통과해야 passes: true인가)

### review-protocol.md
에이전트 간 코드 리뷰 규칙.

포함 내용:
- 리뷰 관점 (구조, 테스트, 보안, 성능)
- 승인 기준 (어떤 조건이면 머지 가능)
- 에스컬레이션 기준 (사람에게 넘기는 경우)

## 선택 파일

### design/[domain].md
도메인별 설계 문서. 데이터 모델, API 설계, 비즈니스 규칙.
에이전트가 해당 도메인 작업 시 참조.

### exec-plans/tech-debt-tracker.md
기술 부채 목록과 우선순위.

```markdown
## 기술 부채 추적

| ID | 영역 | 설명 | 우선순위 | 상태 |
|----|------|------|---------|------|
| TD-001 | billing | 결제 SDK 직접 호출 → Service 래핑 필요 | 높음 | 미착수 |
| TD-002 | shared | 중복 날짜 유틸 통합 | 중간 | 진행중 |

최종 업데이트: YYYY-MM-DD
```

### exec-plans/[plan-name].md
특정 작업의 실행 계획. 에이전트가 복잡한 작업 전에 참조.

## 문서 유지보수 원칙

1. docs/는 코드와 함께 버전 관리 (git)
2. 에이전트가 코드 변경 시 관련 docs/도 함께 업데이트
3. 주기적 "doc-gardening": 오래된 문서 확인 → 업데이트 PR
4. quality-grades.md는 N회 세션마다 또는 스프린트마다 갱신
5. 교차 링크: 문서 간 참조를 명확히, 깨진 링크 방지
